# AI Household Account

자연어로 입력한 소비 내역을 분석하여 거래 데이터를 자동으로 분류하고, 월별 소비 패턴을 시각화해주는 개인 가계부 서비스입니다.

사용자는 `메가커피 4500`, `스벅 6200` 같은 간단한 입력이나 결제 문자 형태의 텍스트를 입력할 수 있으며, 서비스는 금액과 가맹점명을 파싱한 뒤 로컬 규칙과 AI 보정을 통해 카테고리를 분류합니다.

## 주요 기능

### 사용자 인증

Supabase Authentication을 이용하여 회원가입과 로그인을 구현했습니다.

#### 구현 내용

- 이메일 회원가입
- 로그인
- 로그아웃
- 로그인 후 대시보드 접근
- 인증 사용자 기준 거래 데이터 조회

---

### 자연어 거래 입력

사용자가 간단한 문장으로 거래 내역을 입력하면 금액과 가맹점 정보를 추출하도록 구현했습니다.

#### 구현 내용

- `메가커피 4500` 형태의 빠른 입력 지원
- 여러 거래를 한 번에 입력하는 방식 지원
- 결제 문자 원문 기반 거래 후보 추출
- 잘못된 입력 라인 검증
- 거래 금액, 가맹점, 메모, 원문 데이터 분리

---

### 로컬 규칙 기반 카테고리 분류

자주 등장하는 가맹점과 키워드는 AI 호출 없이 로컬 규칙으로 먼저 분류하도록 구현했습니다.

#### 구현 내용

- 카페, 식비, 교통, 생활, 쇼핑, 콘텐츠, 구독, 의료, 수입, 기타 카테고리 정의
- 스타벅스, 메가커피, 맥도날드, 편의점, 지하철 등 키워드 기반 분류
- 로컬에서 확정 가능한 항목은 confidence 값을 높게 처리
- AI 호출 대상은 `기타`로 남은 항목만 선별

---

### AI 카테고리 보정

로컬 규칙으로 분류하기 어려운 거래만 OpenRouter API로 보정하도록 구현했습니다.

#### 구현 내용

- Next.js API Route에서 AI 요청 처리
- OpenRouter Provider 구현
- JSON 응답만 반환하도록 프롬프트 설계
- AI 응답 파싱 및 보정
- 응답 실패 시 로컬 후보 데이터로 fallback 처리

---

### 거래 저장 및 조회

분석된 거래 데이터를 Supabase에 저장하고, 저장된 데이터를 다시 조회해 대시보드와 거래내역 화면에 표시했습니다.

#### 구현 내용

- 로그인 사용자 기준 거래 저장
- 거래 목록 조회
- 거래 데이터 정규화
- 저장 후 거래 목록 재조회
- 월별 거래내역 필터링
- 거래내역 페이지네이션

---

### Dashboard 소비 분석

저장된 거래를 기반으로 이번 달 소비 흐름을 분석하는 대시보드를 구현했습니다.

#### 구현 내용

- 총 지출 / 총 수입 요약
- 카테고리별 지출 합산
- 상위 5개 카테고리 추출
- 도넛 차트 형태의 소비 비중 시각화
- 가장 많이 소비한 분야 기반 소비 인사이트 제공

## 기술적 구현 포인트

### AI Provider 분리

AI 요청 로직을 직접 컴포넌트에 넣지 않고 Provider Layer로 분리했습니다.

#### 구현 내용

- `AIProvider` 인터페이스 정의
- `OpenRouter Provider` 구현
- `Mock Provider` 구현
- 환경변수 `AI_PROVIDER`에 따라 Provider 선택
- 서버 전용 코드에 `server-only` 적용

#### 결과

AI 서비스가 변경되더라도 화면 코드가 아니라 Provider 구현만 교체하면 되는 구조로 만들었습니다.

---

### 로컬 규칙과 AI 보정의 하이브리드 구조

모든 거래를 AI로 보내는 대신, 먼저 로컬 규칙으로 처리하고 애매한 항목만 AI로 보정하도록 설계했습니다.

#### 구현 내용

- 자주 쓰는 가맹점 키워드 로컬 매핑
- 기존 저장 거래를 기반으로 가맹점-카테고리 매핑 재사용
- `기타`로 남은 항목만 AI 요청
- AI 실패 시에도 로컬 후보 데이터 저장 가능

#### 결과

AI 응답 속도와 실패 가능성에 덜 의존하면서도, 필요한 경우에는 AI 보정을 사용할 수 있는 구조를 만들었습니다.

---

### AI 응답 안정성 처리

AI 응답은 항상 안정적인 JSON으로 온다고 보장할 수 없기 때문에 응답 파싱과 fallback 로직을 추가했습니다.

#### 구현 내용

- JSON 객체 추출
- 불필요한 텍스트 제거
- 잘못된 쉼표나 객체 연결 형태 보정
- 응답 timeout 처리
- 빈 응답 처리
- JSON 파싱 실패 시 로컬 후보 반환

#### 결과

AI 응답이 실패하거나 형식이 깨져도 서비스 전체 흐름이 중단되지 않도록 개선했습니다.

---

### Dashboard 로직 분리

Dashboard 페이지에 모든 로직을 직접 작성하지 않고 Hook, Service, Utils, Component로 책임을 나누었습니다.

#### 구현 내용

- `useDashboardTransactions` : 사용자 및 거래 데이터 관리
- `useTransactionAnalysis` : 거래 입력 분석 흐름 관리
- `services.ts` : Supabase 및 API 요청 처리
- `utils.ts` : 금액 포맷, 날짜 처리, 카테고리 분석 유틸 분리
- Dashboard 컴포넌트 분리

#### 결과

Dashboard 페이지는 화면 조합에 집중하고, 데이터 처리와 분석 로직은 별도 모듈에서 관리하도록 개선했습니다.

## 트러블 슈팅

### AI 분석 결과 정확도 문제

#### 문제

초기에는 AI가 간단한 가맹점명도 `기타`로 분류하거나, 응답 형식이 일정하지 않은 문제가 있었습니다.

#### 해결

- 카테고리 기준을 프롬프트에 명확하게 작성
- 로컬 키워드 기반 분류 규칙 추가
- AI는 애매한 항목만 보정하도록 변경
- 응답 JSON 파싱 및 fallback 로직 추가

#### 결과

자주 입력되는 거래는 빠르게 로컬에서 처리하고, AI 실패 상황에서도 거래 저장 흐름이 유지되도록 개선했습니다.

---

### 거래 저장 시 DB 컬럼 불일치 문제

#### 문제

Supabase `transactions` 테이블 구조가 변경되거나 일부 컬럼이 없는 경우 insert 요청이 실패할 수 있었습니다.

#### 해결

- insert 실패 메시지에서 누락 컬럼을 추출
- 없는 컬럼을 제외한 payload로 재시도
- 최대 8회까지 fallback insert 처리

#### 결과

테이블 구조가 일부 달라도 핵심 거래 데이터는 저장될 수 있도록 안정성을 높였습니다.

---

### Dashboard 로직 복잡도 증가

#### 문제

거래 조회, 분석, 저장, 통계 계산, UI 상태 관리가 Dashboard 페이지에 집중되면서 유지보수가 어려워질 가능성이 있었습니다.

#### 해결

- 거래 데이터 관리는 `useDashboardTransactions`로 분리
- 분석 흐름은 `useTransactionAnalysis`로 분리
- Supabase 요청은 `services.ts`로 분리
- 날짜/금액/통계 계산은 `utils.ts`로 분리
- UI는 컴포넌트 단위로 분리

#### 결과

페이지 컴포넌트는 화면 구성에 집중하고, 기능별 로직은 각 파일에서 관리할 수 있게 되었습니다.
